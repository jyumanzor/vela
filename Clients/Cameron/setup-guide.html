Waypoint Two

Set Up Claude Code

Claude Code is your primary writing environment. It runs in your terminal, directly alongside your files. It reads your reference documents, follows your rules, and maintains a log of what happened across sessions.

Install it:

Terminal 
npm install -g @anthropic-ai/claude-code

Or if you prefer Homebrew: brew install claude-code

Verify it works:

Terminal 
claude --version

When Claude Code starts, it automatically reads your CLAUDE.md file and every skill in the skills/ folder. No manual configuration. The rules and methodology load every time.

03
Waypoint Three

Set Up Codex

Codex is your second pair of eyes. While Claude Code is your primary writer and researcher, Codex runs in the background as an adversarial reviewer. It catches what the first model misses.

Get access:

Setup 
npm install -g @openai/codex

Point it at your project:

Terminal 
cd ~/white-papers && codex
● Key difference from Claude Code

Codex runs tasks in the background. You can ask it to review a section while you continue writing in Claude Code. It reads the same project folder and the same skills files.

● Sharing rules between tools

Your CLAUDE.md and skills/ folder work for both tools. Codex reads markdown instruction files the same way Claude Code does. One workspace, two models, zero duplication.

When to use Codex:

After writing a section in Claude Code, open a second terminal and ask Codex: "Read output/introduction.md. Find logical gaps, uncited claims, and arguments that wouldn't survive peer review." Let it work while you move to the next section.

04
Waypoint Four

Create Your Workspace

Your workspace structure determines how well the agent can think. A flat folder of files is chaos. A structured project is a mind the agent can navigate.

Run this to create everything at once:

Terminal 
mkdir -p ~/white-papers/{ref-docs/{papers,data,notes,drafts},output/_archive,skills}

This creates:

white-papers/
├── CLAUDE.md ← your rules (installed in Step 6)
├── _BUILD_LOG.md ← session history
├── _WORKSPACE.md ← canonical files, archive rules
├── skills/ ← portable methodology (13 files)
├── ref-docs/ ← source materials
│   ├── papers/
│   ├── data/
│   ├── notes/
│   └── drafts/
├── output/ ← current canonical drafts
└── output/_archive/ ← dated previous versions

The output/_archive/ folder is important. Every time you have a draft worth keeping, date it and move it there. Your output/ folder should only ever contain the current canonical version.

05
Waypoint Five

Organize Reference Documents

Your agent is only as good as what it can read. If a study isn't in the ref-docs folder, the agent cannot cite it. If a dataset isn't there, the agent cannot analyze it. There is no magic.

● papers/

Academic papers, published research, reports from institutions. The things you'd put in a bibliography. PDFs, downloaded articles, anything with an author and a date.

● data/

Datasets, spreadsheets, CSVs, raw numbers. If your paper makes a quantitative claim, the data supporting it lives here.

● notes/

Your personal notes, interview transcripts, observations, rough thinking. The messy stuff that isn't published but informs your argument.

● drafts/

Previous drafts and reviewer feedback. When a reviewer sends you comments, put them here. The agent can read reviewer feedback and help you address it point by point.

For example: if you're writing about AI adoption in healthcare, your ref-docs/papers/ should have the actual studies you're citing. Not summaries. Not abstracts. The full papers.

Waypoint Six

Install Your Rules

CLAUDE.md is the file that tells Claude Code how to behave in your project. Without it, the agent is generic. With it, the agent knows your citation standards, your operating rhythm, and what to check before you submit.

Your starter kit includes CLAUDE.md, two workspace templates, and thirteen skill files. Download them individually or grab the full kit from the hero section above.

Your thirteen skill files:

operating-loop.md The BUILD → CHALLENGE → RESOLVE → PROMOTE rhythm for every draft section
testifying-expert.md Provenance categories (PUBLISHED/DERIVED/ANALYST) and defensibility checks
cross-model-review.md How to use a second AI model as an adversarial reviewer
plausible-but-wrong-numbers.md Catching AI-fabricated statistics that sound right but aren't
ai-data-smoothing.md Catching AI that interpolates volatile data into smooth trends
silent-data-drop.md Catching data that gets filtered away unnoticed during analysis
build-log-protocol.md Cross-session memory so you never lose context between writing sessions
workspace-hygiene.md Folder structure rules that prevent version confusion
operational-inheritance.md How to promote durable learnings from one project into your permanent system
claude-handoff-notes.md How to brief an agent on a project so it starts productive, not confused
holistic-review.md Drift detection after multiple edit rounds to catch accumulated inconsistencies
testing-ai-output.md Validation checkpoints, forbidden strings, oracle validation for AI-generated content
multi-round-editing.md Handling revision cycles without losing earlier decisions or introducing regressions

You don't need to memorize these. The agent reads them automatically before every major action. They're the encoded judgment that makes the difference between a generic chatbot and a writing environment that catches its own mistakes.

Waypoint Seven

Your First Session

Start Claude Code from your project folder:

Terminal 
cd ~/white-papers && claude

When Claude starts, it reads your CLAUDE.md and skills automatically. No configuration needed. The operating loop drives every section you write:

Build

Write the section draft

Challenge

Adversarial review

Resolve

Accept, reject, revise

Promote

Record in build log

A realistic first session looks like this:

1

You type: "Write the introduction section of my paper on AI adoption in healthcare. Read the papers in ref-docs/ first."

2

The agent reads your reference documents, understands the landscape, and writes a draft introduction with citations pointing to specific papers.

3

You type: "Review this section. Are all claims cited? Any uncited statistics? Check provenance on every number."

4

The agent identifies weak spots: an uncited statistic in paragraph 2, a claim that needs a DERIVED tag, a sentence that sounds confident but has no backing source.

5

Meanwhile, in a second terminal: "codex: Read output/introduction.md and find logical gaps I missed." Codex runs in the background while you fix the issues Claude found.

6

You type: "Fix those issues. Then append to _BUILD_LOG.md what we did this session."

That's it. BUILD, CHALLENGE, RESOLVE, PROMOTE. Every section, every session. The build log means you never start cold next time.

08
Waypoint Eight

Before You Submit

Every claim in your paper falls into one of three provenance categories. Tagging them isn't bureaucracy. It's how you know what's solid, what's derived, and what's your judgment call.

PUBLISHED
Directly from a citable source. Paper, dataset, institutional report.
"42% of hospitals adopted AI tools by 2024 (McKinsey, 2024)"
DERIVED
Calculated from published data. Show the math.
"Adoption grew 3.2x from 2020 to 2024 (calculated from Table 2)"
ANALYST
Your judgment call. Flag it explicitly, defend it in the text.
"This trend suggests a tipping point by 2026 (author's assessment)"

Cross-model verification: Before submitting, run your core argument through Codex as a final check. Paste your thesis and key claims. Ask it to find weaknesses, uncited claims, or logical gaps. What survives two models disagreeing is what you can trust.

Your pre-submission checklist:

✓
Every cited number has a provenance tag (PUBLISHED / DERIVED / ANALYST)
✓
Cross-model adversarial review completed on full draft
✓
No uncited statistics remain
✓
Build log reflects all major revision decisions
✓
Reference docs folder is complete and organized
✓
Holistic review run for drift after multiple edit rounds
✓
PDF export reviewed for formatting quality
